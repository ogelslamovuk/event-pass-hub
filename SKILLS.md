# skill.md — Правила постановки задач для Codex при минорных правках без локального проекта

## 1. Когда применять

Этот skill применять, когда задача относится к мелким или средним правкам в проекте через Codex без локального проекта и без полноценной ручной проверки до merge/deploy.

Типовые задачи:
- русификация UI;
- замена видимых текстов;
- добавление tooltip / HelpTooltip;
- мелкие UX/UI-правки;
- правки одного экрана или одного блока;
- точечные изменения JSX/TSX;
- исправление wording/copy;
- добавление подсказок к карточкам, кнопкам, input/select/textarea;
- визуальные правки без изменения бизнес-логики.

Не применять как основной режим для:
- backend-логики;
- сложных архитектурных изменений;
- изменения store/localStorage/API;
- миграций данных;
- больших refactor-задач;
- задач, где есть локальный проект и можно быстро тестировать руками.

Главный принцип:
Codex — исполнитель. Он не принимает продуктовые, UX, архитектурные или текстовые решения.

---

## 2. Базовая установка перед написанием ТЗ

Перед выдачей ТЗ Codex нужно сформировать задачу так, чтобы у него не было пространства для интерпретации.

Запрещены формулировки:
- «найди и исправь»;
- «добавь где нужно»;
- «аналогично»;
- «по смыслу»;
- «везде»;
- «если есть» без точного правила;
- «улучши»;
- «приведи к единому виду»;
- «проставь tooltip по всем элементам»;
- «сделай красиво»;
- «посмотри сам».

Разрешены формулировки:
- файл;
- точный элемент;
- точный текущий текст;
- точный новый текст;
- точный паттерн JSX;
- точное место вставки;
- точный список запретов;
- точная самопроверка;
- точный acceptance checklist.

Если точного текущего кода нет, сначала нужно запросить актуальный файл/diff/скрин/ссылку и не писать ТЗ с догадками.

---

## 3. Scope должен быть максимально узким

В начале каждого ТЗ обязательно указать:

```text
Работать от текущего main.
Менять только:
- path/to/file.tsx

Не трогать:
- список файлов
- routes
- store
- localStorage
- selectors
- types
- business logic
- onClick
- onChange
- фильтрацию
- сортировку
- соседние страницы

PR не делать.
Commit не делать.
Вернуть только один чистый unified diff/patch.
```

Если задача касается одного блока, нельзя давать весь модуль целиком.

Если задача уже провалилась один раз, следующая сессия Codex должна начинаться с нуля. Не просить Codex «дочинить» плохой patch, если он начал дублировать код или расширять scope.

---

## 4. Codex нельзя просить принимать решения

Codex не должен:
- выбирать, где нужен tooltip;
- придумывать текст tooltip;
- решать, что переводить;
- решать, какие элементы важные;
- определять, что является карточкой;
- менять структуру для удобства;
- улучшать UX сверх задачи;
- добавлять helper-компоненты без прямого указания;
- переименовывать вкладки, если это не указано явно;
- трогать соседние страницы.

Если элемент не описан в ТЗ, Codex обязан его не трогать.

---

## 5. Для UI/tooltip задач использовать видимые HelpTooltip, а не HTML title

Правило:
- HTML `title` не использовать как основной tooltip.
- Для видимого tooltip использовать существующий `HelpTooltip`.
- Если в проекте уже есть UX-паттерн с иконкой подсказки — повторять его.
- Не смешивать `title` и `HelpTooltip` на одном элементе.
- Если рядом уже есть `HelpTooltip`, не добавлять `title`.

Паттерн для карточки/плитки/KPI:

```tsx
function CardHelp({ text }: { text: string }) {
  return (
    <div className="absolute right-4 top-4 z-10">
      <HelpTooltip text={text} />
    </div>
  );
}
```

```tsx
<div className="relative">
  <button/div ...>
    ...
  </button/div>
  <CardHelp text="Точный текст подсказки." />
</div>
```

Важно:
- `HelpTooltip` не вкладывать внутрь `button`.
- Для кликабельной карточки wrapper — `div.relative`, кнопка внутри, tooltip соседним элементом.
- У кнопки сохранить `onClick`, `style`, содержимое и текущие классы, кроме точечно указанного `w-full/pr-9/relative`.

Паттерн для обычной кнопки/ссылки:

```tsx
<div className="inline-flex items-center gap-1">
  <button/link ...>Текст</button/link>
  <HelpTooltip text="Точный текст подсказки." />
</div>
```

Паттерн для input/select/textarea:

```tsx
<div className="relative">
  <input/select/textarea className="... pr-9 ..." ... />
  <div className="absolute right-2 top-1/2 -translate-y-1/2">
    <HelpTooltip text="Точный текст подсказки." />
  </div>
</div>
```

Для textarea обычно использовать:

```tsx
<div className="absolute right-2 top-3">
  <HelpTooltip text="Точный текст подсказки." />
</div>
```

## 6. Таблицы

По умолчанию tooltip не ставить:
- на table;
- на tr;
- на td;
- на th;
- на заголовки колонок;
- на SortableHeader;
- на обычные значения в таблице.

Разрешено ставить tooltip только на action-кнопки внутри таблиц, если это явно указано:
- Просмотр;
- Продолжить;
- Доработать;
- Одобрить;
- Вернуть на доработку;
- Отклонить.

---

## 7. Главное правило против дублей

Codex часто ошибается так:
- добавляет новую строку рядом со старой;
- добавляет новый JSX-блок рядом со старым;
- дублирует массив;
- дублирует кнопку;
- делает 12 KPI вместо 6;
- оставляет старый бренд и добавляет новый ниже.

Поэтому в ТЗ всегда писать:

```text
ВАЖНО:
Эта задача должна быть выполнена точечными заменами.
Нельзя добавлять новые строки рядом со старыми.
Нельзя дублировать JSX.
Нельзя дублировать массивы.
Нельзя дублировать кнопки.
Старый элемент должен быть удалён или заменён, а не оставлен рядом.
```

Если меняется массив, лучше требовать:
- «заменить весь массив целиком»;
- указать полный итоговый массив;
- указать ожидаемое количество объектов.

Если меняется бренд/лейбл, требовать:
- «старый текст должен отсутствовать»;
- «новый текст должен встречаться ровно N раз».

## 8. Когда использовать Python-скрипт в ТЗ

Если Codex уже ошибался с дублированием или задача содержит много точечных замен, лучше не описывать изменения абстрактно, а дать Codex временный Python-скрипт:

1. Создать `apply_..._patch.py`.
2. В скрипте использовать `replace_once`.
3. Если вхождений не ровно 1 — падать.
4. Запустить скрипт.
5. Удалить скрипт.
6. Запустить self-check.
7. Запустить build.

Базовая функция:

```python
def replace_once(text: str, old: str, new: str, label: str) -> str:
    count = text.count(old)
    if count != 1:
        raise SystemExit(f"{label}: expected exactly 1 occurrence, found {count}")
    return text.replace(old, new, 1)
```

Это снижает риск, что Codex добавит новый блок рядом со старым.

## 9. Self-check обязателен

Для Codex-задач без локального проекта self-check обязателен.

Self-check должен проверять:
- список изменённых файлов;
- отсутствие старых строк;
- точное количество новых строк;
- отсутствие дублей;
- отсутствие запрещённых import;
- отсутствие запрещённых JSX;
- отсутствие `title=`, если HTML title запрещён;
- количество KPI/объектов/кнопок;
- отсутствие временного скрипта;
- build.

Пример структуры:

```text
Обязательная самопроверка:

1. Проверить git diff --name-only.
2. Проверить, что изменены только разрешённые файлы.
3. Проверить, что старые строки отсутствуют.
4. Проверить, что новые строки встречаются ровно нужное количество раз.
5. Проверить, что нет дублей.
6. Проверить, что нет запрещённых import/JSX/title.
7. Запустить npm run build.
```

Если self-check падает, Codex должен исправить только в рамках разрешённых файлов.

---

## 10. Acceptance должен быть исполнимым, не философским

Acceptance писать как чеклист фактов.

Плохо:

```text
UI стал понятнее.
Tooltip проставлены везде.
```

Хорошо:

```text
1. Изменён только src/components/admin/AdminDashboard.tsx.
2. kpiCards содержит ровно 6 объектов.
3. Каждый KPI-объект имеет tooltip.
4. Строка "Все заявки →" встречается ровно один раз.
5. Строка "Все операции →" встречается ровно один раз.
6. HTML title не используется.
7. HelpTooltip не вложен внутрь button.
8. npm run build проходит.
```

## 11. Если Codex прислал плохой patch

Плохой patch — это patch, где есть:
- старый и новый текст одновременно;
- дубль массива;
- дубль JSX-блока;
- дубль кнопки;
- изменение файлов вне scope;
- изменение логики;
- изменение меню, если меню запрещено;
- build не проходит;
- self-check не проходил;
- Codex «дочинил» через новые костыли.

Действие:
- не просить «поправь этот diff»;
- откатить patch полностью;
- начать новую чистую сессию Codex;
- переписать ТЗ так, чтобы исключить конкретную ошибку;
- по возможности дать replace_once-скрипт и self-check.

---

## 12. Формат хорошего ТЗ

Оптимальная структура:

```text
Задача для Codex:
[одно предложение]

Проект:
[repo]

Работать от текущего main.

Менять только:
- file1
- file2

PR не делать.
Commit не делать.
Вернуть только один чистый unified diff/patch.

1. Главная задача
[точно что меняем]

2. Что запрещено
[список запретов]

3. Точные изменения
[файл → элемент → было → стало]

4. Паттерн реализации
[если нужны tooltip/card/helper]

5. Самопроверка
[python/bash проверки]

6. Acceptance
[проверяемые факты]
```

## 13. Особое правило для бренда и русификации

Для текстовых замен всегда писать:

```text
Старые строки должны быть удалены, а не оставлены рядом.
Новые строки должны встречаться ровно указанное количество раз.
```

Для брендов всегда указывать регистр буквально:
- `Центр Управления`
- `by Cinemalab`
- не `Buy Cinemalab`;
- не `By Cinemalab`;
- не `Центр управления`, если утверждено другое.

---

## 14. Особое правило для меню

Если пользователь говорит «не трогать меню», это означает:
- не менять labels;
- не менять tabTitles;
- не менять JSX;
- не добавлять HelpTooltip;
- не добавлять title;
- не менять padding/classes;
- не добавлять wrapper;
- не добавлять `menuTooltip`;
- не менять структуру массива.

Даже если рядом есть нужный текст, меню остаётся вне scope.

---

## 15. Особое правило для задач без локального проекта

Поскольку пользователь не может быстро проверить UI до merge/deploy:
- нельзя полагаться на «визуально должно быть нормально»;
- нужно проверять diff структурно;
- build обязателен;
- acceptance должен позволять принять/отклонить patch по diff;
- ручную проверку формулировать только как post-merge/post-deploy validation.

---

## 16. Рабочая формула

Для Codex-задач такого типа думать так:

```text
Не объяснить Codex, что мы хотим.
А ограничить Codex так, чтобы он физически не мог сделать не то.
```

Codex не должен «понять задачу».
Codex должен выполнить точные замены и пройти self-check.
